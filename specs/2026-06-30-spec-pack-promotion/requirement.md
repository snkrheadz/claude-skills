## 背景 / 解くべき課題
spec 駆動パイプライン（リポジトリ監査からマージ可能な変更までを、各段で人間ゲートを挟みつつ進める一連の手順）は、設計上どのリポジトリにも持ち込めることを意図している。しかし現状は個人の dotfiles リポジトリにのみ存在し、そのマシン以外では使えない。一方このマーケットプレイスは「役割別パックを誰でも `/plugin install` で同じ環境を組める」ことを目的としており、パイプラインの本来の価値（どこでも同じ規律を再現する）と完全に一致する。本要件は、このパイプラインをマーケットプレイスの独立したパックとして提供し、任意のリポジトリでインストール・利用できる状態にする。あわせて、パイプラインが内部で前提とする他コマンドへの参照が、インストール後も解決できる形になっていることを保証する。

## スコープ
- in: spec 駆動パイプラインを構成する一連のフェーズ（監査・要件・設計・タスク分解・レビュー）を、このマーケットプレイスの独立したパックとして提供すること。
- in: 各フェーズがマーケットプレイスのパック収録物の規約（選択可能・トリガー記述・モデル指定など、既存パックと同じ品質基準）に適合すること。
- in: パイプラインが内部で参照する他コマンド（実装フェーズや PR 作成・テスト修復など）が、インストール済み環境で解決できること（参照先の提供 or 名前空間の整合）。
- in: パックがマーケットプレイスのカタログに登録され、カタログの整合性チェック（既存の検証手順）を通ること。
- in: パイプラインの利用方法を、人間が読んで理解できるドキュメントとして同梱すること。
- out: パイプライン自体の手順・ゲートの仕様変更（移植であって再設計ではない。各フェーズの振る舞いは現行を踏襲）。
- out: マーケットプレイスのリポジトリ名称変更（別案件。先に昇格、後に改名の順）。
- out: 元のリポジトリ側のローカルコピー削除や参照差し替えの完了（昇格が済んだ後の追従作業。本件は「パックとして入手可能になる」までを対象とし、移行作業は別途）。
- out: パイプライン由来の生成物（specs 配下の各案件成果物）の取り扱いルール整備。

## 受け入れ条件（EARS）
- WHEN 利用者がこのマーケットプレイスから当該パックをインストールしたとき THE SYSTEM SHALL 監査・要件・設計・タスク分解・レビューの各フェーズを、スラッシュコマンドとして選択・実行できる状態にする。
- WHEN いずれかのフェーズを実行したとき THE SYSTEM SHALL 現行パイプラインと同じ成果物（各フェーズの仕様書）と同じ人間ゲートの停止挙動を提供する。
- IF あるフェーズが他コマンド（実装・PR 作成・テスト修復など）を案内・前提とするとき THEN THE SYSTEM SHALL インストール済み環境で解決可能な参照（提供済みの収録物 or 整合した名前空間）を案内する。
- WHEN マーケットプレイスのカタログ整合性チェックを実行したとき THE SYSTEM SHALL 当該パックを含めて成功（エラーなし）を返す。
- WHEN 利用者がパックの使い方を知りたいとき THE SYSTEM SHALL 同梱の人間向けドキュメントから各フェーズの役割・ゲート・利用例を把握できる。

## 非機能要件 / 制約
- 既存パック（core / pm / eng / research / strategy / writing）の挙動・収録物を変更しないこと（追加のみ）。
- パイプラインの各フェーズの振る舞い・ゲート・altitude のルールを現行から変えないこと（移植の忠実性）。
- マーケットプレイスの既存の品質基準（収録物のメタデータ規約、版の整合、カタログ検証）に準拠すること。
- 新たな重い外部依存を持ち込まないこと。
- インストール後、特定のリポジトリ構造に依存せず任意のリポジトリで機能すること（パイプラインの repo 非依存性を維持）。

## 検証方法
- マーケットプレイスから当該パックをインストールし、各フェーズのコマンドが選択・実行でき、現行と同じ成果物と人間ゲートの停止が得られることを確認する。
- パイプラインが参照する他コマンドが、インストール済み環境から実際に解決できることを確認する（参照切れが無い）。
- マーケットプレイスの整合性チェック（既存の検証手順）を実行し、当該パックを含めてエラー無く通ることを確認する。
- 同梱ドキュメントだけを読んで、第三者が各フェーズの役割とゲート操作を理解できることを確認する。
- 既存パックのインストール・利用が本変更後も従来どおり機能することを確認する（回帰なし）。

## 解決済みの前提（Gate ① レビューで確定）
- **実装フェーズ用コマンドは本パックに同梱し、自己完結させる。** パイプラインが内部参照する実装系コマンドは、外部パックや個人リポジトリに依存させず、この spec パックの収録物として含める。これによりパック単体のインストールでパイプラインが端から端まで参照切れなく機能する（受け入れ条件「参照が解決できる」を、外部依存ではなく同梱で満たす）。
- **監査ルーブリック（3本柱）も本パックに同梱し、持ち運べる形にする。** 監査フェーズが用いる評価基準を、どのリポジトリでも同じ基準で適用できるようパックに含める。
- **成果物例の同梱は任意。** 既存案件の仕様書例をドキュメントに載せるかは実装時の判断でよい（必須ではない）。

> 補足: 同梱方針により、本パックの収録物は「監査・要件・設計・タスク分解・実装・レビュー」のフェーズ群＋ルーブリックとなり、パック単体でパイプラインが自己完結する。PR 作成・テスト修復など他パック（eng）が担うコマンドは、パイプラインの案内文が指す参照先として整合していればよく、本パックに重複収録はしない（昇格＝移植であり、既存パックの機能を奪わない）。
